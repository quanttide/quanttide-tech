## 报告二：客户服务章程

### 类型
制度规范文档

### 结构
总则 → 基本原则 → 角色职责 → 权责体系 → 运行机制 → 监督反馈 → 核心底线

### 具体内容

**一、总则（Why）**
1.1 目的：为保障客户服务的稳定性、专业性与可预期性，建立统一服务标准与运行机制，避免服务依赖个人状态，确保业务长期健康发展。
1.2 适用范围：适用于公司所有客户服务相关业务（含课堂BU、数据BU）及全体成员。
1.3 定义：客户服务指从客户接入到最终交付全过程中的响应、沟通、执行与反馈行为的总和。

**二、基本原则**
1. 结果导向原则：以客户实际体验为最终评价标准
2. Owner负责原则：每个客户必须有唯一负责人，对结果与体验负责到底
3. 响应优先原则：先响应，再解决；不得出现客户被忽视
4. 标准化原则：服务必须可复制、可预期，不依赖个人能力波动
5. 异常优先原则：风险与异常优先级高于常规流程
6. 复盘闭环原则：所有问题必须沉淀为规则或流程优化

**三、角色与职责**
原则：职责可协同，责任不可转移
3.1 COO：制定与优化客户服务制度；监督整体执行情况；主导重大异常与机制调整
3.2 秘书长（中台）：日常监控客户状态；协调资源与跨角色问题；跟踪异常处理进度
3.3 Owner（客户负责人）：客户第一责任人；负责响应、推进、结果与关系维护；对所有客户问题负责闭环
3.4 执行人：按标准完成具体交付；对任务质量与进度负责

**四、权责体系**
4.1 决策权：是否接单由负责人/管理层决策；方案调整Owner主导，必要时上报；异常处理Owner先行，重大问题升级
4.2 责任机制：客户体验问题→Owner负责；执行问题→执行人负责；系统问题→COO负责
4.3 追责原则：出现响应超时、交付延误、客户投诉需复盘并明确责任；不允许模糊归因或集体免责

**五、运行机制与流程**
5.1 客户接入：明确需求；指定Owner；建立沟通渠道
5.2 响应机制：工作时间14-45分钟内响应；非工作时间2小时内响应；响应必须包含理解+下一步动作
5.3 服务流程：课堂BU：建群→课前确认→上课→作业→反馈；数据BU：需求→执行→同步→交付→确认

**六、监督与反馈**
6.1 数据监控：必须记录响应时效、是否超时、当前节点、客户状态（🟢🟡🔴）
6.2 状态管理：🟢正常：按计划推进；🟡风险：存在延误可能；🔴异常：已发生问题；优先处理🔴
6.3 检查机制：每日检查异常客户；每周统一复盘（若有）
6.4 反馈机制：所有问题必须转化为流程优化、标准补充、责任明确

**七、核心底线**
- 不允许客户被忽视
- 不允许责任不清
- 不允许服务不稳定

## 报告三：2026-3-19-COO运营复盘

### 类型
运营复盘报告

### 结构
元数据区 → 复盘区 → 附议区 → 决议区 → 代办区

### 具体内容

**元数据区**
- 发起人：@黄梓姗 @张果
- 涉及业务单元：课堂BU
- 参与角色：秘书长层、BU负责人、执行层
- 文档性质：内部复盘 + 制度修订（需附议）
- 目标：统一问题认知，明确责任边界，推动机制落地

**复盘区**
背景：鲍宇航客户消息无人响应并负责

1. 客户响应不及时（直接影响体验）
- 存在未及时回复客户消息的情况
- 响应时间不稳定，依赖个人自觉
- 结果：客户体验波动，信任感下降

2. 权责划分不清（系统性问题）
- 客户问题明确唯一责任人，但负责人培训不到位
- 多人参与但无人对最终结果负责
- 本质问题：缺少"客户Owner机制"，责任未闭环

3. 负责人能力与意识不足（组织问题）
- 负责人未经过系统化培训
- 对"客户体验负责"认知不足
- 结果：执行停留在"完成任务"，而非"对结果负责"

4. COO管理盲区（机制问题）
- 脱离一线后，对问题感知下降
- 当前依赖"汇报"，信息存在失真
- 项目推进被动，依靠上级/极端事件反馈触发
- 本质问题：缺少过程监控与数据化反馈机制

**附议区**
