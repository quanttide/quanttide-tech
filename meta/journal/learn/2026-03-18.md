### 2. 核心框架：记忆分类框架

#### 陈述型记忆 - 九宫格模型
根据时间（过去/现在/未来）和类型（事件类/语义类/自我类）对知识进行分类：

| | 事件类 | 语义类 | 自我类 |
|------|--------|--------|--------|

| **未来** | `profile/` Profile<br>工作档案 | `essay/` Essay<br>工作札记 | Roadmap<br>发展蓝图 |

#### 程序型记忆五层体系
根据约束力和权威性分层：

| 层次 | 英文名称 | 中文名称 | 法律隐喻 | 子模块 |
|------|----------|----------|----------|--------|
| L1 | Paper | 工作论文 | 立法原理 | `paper/` |
| L2 | Usecase | 工作案例 | 精选判例 | `usercase/` |
| L3 | Handbook | 工作手册 | 权威汇编 | `handbook/` |
| L4 | Specification | 工程标准 | 程序性法律 | `specification/` |

## 学习发现：meta/bylaw 层次结构问题

### 当前问题
在 quanttide-tech 仓库中，发现 `meta/bylaw` 目录存在层次感不足的问题：
- `journal.md`：通用日志规范（父级）
- `learn_journal.md`：学习日志专用规范（子级）

两者平级存放，未体现"通用→专用"的层级关系。

### 对比 quanttide-founder
quanttide-founder 的 `journal` 目录使用**子目录分类**：
```
journal/
├── write/      # 写作日志
├── code/       # 代码日志
└── default/    # 默认日志
```

这种模式可以应用到 `meta/bylaw` 目录：
```
meta/bylaw/
├── journal.md          # 通用日志规范（父级）
└── learn/              # 学习日志专用规范（子目录）
    └── journal.md      # 或 keep_journal.md
```

### 优雅的解决方案
基于 quanttide-founder 的"子目录分类"习惯，提出以下方案：

**方案1：子目录分类（推荐）**
```
meta/bylaw/
├── journal.md          # 通用日志规范
└── learn/
    └── journal.md      # 学习日志专用规范
```

优点：
- 体现层级关系（learn 是 journal 的子集）
- 符合 quanttide-founder 的命名习惯
- 便于扩展其他日志类型（如 code, write 等）

**方案2：文件命名体现层级**
```
meta/bylaw/
├── 01_journal.md           # 通用日志规范
└── 02_learn_journal.md     # 学习日志专用规范
```

优点：
- 简单直接
- 编号体现层级关系

缺点：
- 不符合 quanttide-founder 的子目录分类习惯

**推荐方案1**，因为它更符合 quanttide-founder 的命名习惯，且更具扩展性。

## 学习发现：为什么不定义为 bylaw

### 语义层级不匹配
**bylaw 的定义（L5）：**
- 法律隐喻：宪法
- 约束力：最高级别
- 内容：工作章程、基本规则

**日志规范的实际情况：**
- 日志规范是**具体的操作指南**，不是最高级别的章程
- 日志规范应该属于**具体实现层面**，而不是**宪法层面**

### 参考 quanttide-founder 的结构
在 quanttide-founder 中：
- `meta/` 目录非常简单，只有元数据文件
- 没有复杂的 `meta/bylaw/` 结构
- 日志规范放在 `journal/README.md` 中

这表明：**日志规范不是 bylaw，而是目录说明**。

### 程序型记忆五层体系的适用范围
重新审视五层体系：
| 层次 | 名称 | 法律隐喻 | 适用内容 |
|------|------|----------|----------|
| L1 | Paper | 立法原理 | 理论、原理 |
| L2 | Usecase | 精选判例 | 案例、示例 |
| L3 | Handbook | 权威汇编 | 手册、指南 |
| L4 | Specification | 程序性法律 | 工程标准 |
| L5 | Bylaw | 宪法 | 工作章程 |

**日志规范应该属于哪一层？**
- 不是 L1（理论）
- 不是 L2（案例）
- 可能是 L3（手册）或 L4（标准）

但**不放入五层体系**，而是放在 `meta/journal/` 目录中更合适。

### 简化结构 vs 复杂结构
**如果定义为 bylaw：**
```
meta/bylaw/
├── journal.md           # 工作章程：日志规范
└── learn/               # 学习日志专用规范
    └── journal.md
```

**实际操作（journal 方案）：**
```
meta/journal/
├── README.md           # 日志规范主说明
└── learn/              # 学习日志
    └── README.md       # 学习日志规范说明
```

**简化理由：**
1. 避免复杂的层级体系
2. 日志规范是**具体操作指南**，不是**最高章程**
3. 通过目录结构自然体现层级关系

### 功能导向 vs 体系导向
- **bylaw 方案**：体系导向，强行纳入五层体系
- **journal 方案**：功能导向，按实际功能组织

**核心思想：**
> 日志规范是**具体的操作指南**，应该放在对应的日志目录中，而不是作为最高级别的 bylaw。
